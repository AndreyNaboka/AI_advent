# AI-ассистент службы поддержки

## Архитектура

`support_service.py` объединяет два источника контекста:

1. RAG индексирует FAQ и продуктовую документацию из `support_kb/`;
2. `MCPCRMClient` получает профиль пользователя и тикет из read-only JSON CRM
   через `mcp_crm_server.py`.

На вход передаются `ticket_id` и вопрос. Ассистент добавляет к поисковому запросу
категорию тикета, диагностический код, теги и безопасные сигналы CRM. Поэтому
одинаковый пользовательский вопрос может получить разные ответы в зависимости
от конкретного тикета.

В ответ не включаются имя и email пользователя. Выводятся только ID, тариф,
статус аккаунта, локаль и разрешённые диагностические сигналы.

## Тестовые сценарии

В `support_data/crm.json` находятся полностью искусственные записи:

- `TCK-1001` — аккаунт заблокирован после семи неудачных попыток;
- `TCK-1002` — новый пользователь не подтвердил email;
- `TCK-1003` — домен пользователя не совпадает с настройками SSO.

Один и тот же вопрос для разных тикетов:

```bash
venv312/bin/python support_service.py \
  --ticket TCK-1001 \
  --question "Почему не работает авторизация?"

venv312/bin/python support_service.py \
  --ticket TCK-1002 \
  --question "Почему не работает авторизация?"

venv312/bin/python support_service.py \
  --ticket TCK-1003 \
  --question "Почему не работает авторизация?"
```

Проверьте, что ответы соответственно говорят о временной блокировке, подтверждении
email и обновлении SSO-домена. Каждый ответ должен содержать разделы «Контекст CRM
через MCP», «Ответ поддержки» и «Источники RAG».

## Интерактивный режим

```bash
venv312/bin/python support_service.py
```

Сервис покажет список тикетов. Команда `/ticket TCK-1002` переключает активный
тикет, `/exit` завершает диалог.

## HTTP-сервис

Запуск на локальном интерфейсе:

```bash
venv312/bin/python support_service.py --serve --port 8090
```

Проверка состояния:

```bash
curl http://127.0.0.1:8090/health
```

Запрос поддержки:

```bash
curl -X POST http://127.0.0.1:8090/support \
  -H 'Content-Type: application/json' \
  -d '{"ticket_id":"TCK-1001","question":"Почему не работает авторизация?"}'
```

## MCP CRM

Сервер предоставляет инструменты:

- `get_user` — получить пользователя по `user_id`;
- `get_ticket_context` — получить тикет и связанного пользователя;
- `list_tickets` — показать доступные тикеты.

По умолчанию читается `support_data/crm.json`. Другой файл можно указать через
переменную окружения `AI_ADVENT_CRM_FILE`. MCP-сервер ничего не изменяет в CRM.

## Автоматические тесты

```bash
venv312/bin/python -m unittest tests.test_support_service -v
```

Тесты проверяют MCP handshake, получение тикета, RAG, разные ответы на один
вопрос, фильтрацию персональных данных и настоящий HTTP-запрос к `/support`.
